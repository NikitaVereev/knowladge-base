Ansible — это инструмент автоматизации IT-инфраструктуры (Configuration Management). Он позволяет описывать состояние серверов в виде кода (Infrastructure as Code) и автоматически приводить их к этому состоянию.

## Ключевые особенности

### 1. Agentless (Безагентный)
Ansible не требует установки специального ПО (агентов) на управляемые серверы. Всё, что нужно — это **SSH** и **Python** (который есть почти везде).
* **Плюс:** Легкий старт, меньше нагрузки на серверы, меньше уязвимостей.
* **Минус:** Медленнее на тысячах хостов по сравнению с агентскими системами (Chef/Puppet), но это решается масштабированием контроллеров.

### 2. Декларативный подход (Идемпотентность)
Вы описываете *желаемое состояние* системы, а не *набор команд*.
* **Пример:** "Пакет nginx должен быть установлен".
* **Работа:** Если пакета нет — Ansible его установит. Если пакет уже есть — Ansible ничего не сделает (вернет статус `ok`).
Это свойство называется **идемпотентностью**: повторный запуск playbook безопасен и не сломает уже настроенный сервер.

### 3. Простота (YAML)
Конфигурации (Playbooks) пишутся на YAML, который легко читается даже без знания программирования. Это позволяет разработчикам и админам говорить на одном языке.

## Сравнение с аналогами

| Инструмент | Тип | Агент | Язык | Назначение |
| :--- | :--- | :--- | :--- | :--- |
| **Ansible** | Config Mgmt | Нет (SSH) | YAML | Настройка серверов, деплой приложений, оркестрация |
| **Terraform** | Provisioning | Нет (API) | HCL | Создание инфраструктуры (AWS, VM, сети) |
| **Chef/Puppet** | Config Mgmt | Да | Ruby/DSL | Сложные Enterprise среды со строгим контролем состояния |

> **Итог:** Используйте **Terraform** для создания виртуальных машин (железа), а **Ansible** — для их настройки и установки софта внутри.
